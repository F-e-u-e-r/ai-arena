# 非洲大草原生態系統（Canvas 2D）— Sonnet 5 · max effort

## 如何啟動

單一 self-contained `index.html`，直接用瀏覽器開啟，或在本目錄跑一個 static server：

```bash
python3 -m http.server 8000
# 開啟 http://localhost:8000/
```

不需要建置流程、不需要安裝任何套件，沒有使用任何外部函式庫或框架。

## 為何選擇 Canvas

題目本身要求「禁止使用任何框架、遊戲引擎、外部函式庫」且「必須使用 Canvas 2D API」，這已經直接決定了技術選型。Canvas 2D 相對 SVG／DOM 節點的優勢在這題特別明顯：

- 動物身體是用十幾個 `ellipse` / `arc` / 二次貝茲曲線路徑即時畫出來的，加上最多 450 個粒子，若用 DOM 節點（例如每個粒子一個 `<div>`）會有大量 reflow／repaint 成本；Canvas 每幀只是重繪像素，沒有 DOM tree 更動的開銷。
- 天氣切換、雷擊閃光、地面積水這類「整個畫面短暫改變」的效果，用 `fillRect` 疊一層半透明色即可，不需要維護額外的圖層元素。
- 所有繪製都在同一個 2D context 裡順序發生，繪製順序即是視覺疊放順序（先畫天空、再畫地景、再畫動物、最後畫粒子與閃電），不需要額外管理 z-index。

## 架構總覽

| 類別 / 模組 | 職責 |
| --- | --- |
| `Animal`（基底） + `Lion` / `Elephant` / `Leopard` | 共用的移動、景深縮放、跟隨隊形、龍捲風逃跑邏輯；各自覆寫 `updateState()` 定義專屬狀態機與 `draw()` 繪製 |
| `WeatherManager` | 6 種天氣單選、粒子生成速率、天氣對移動速度的乘數、切換時的粒子回收邏輯 |
| `SeasonManager` | 乾季／濕季，每 60 秒自動輪轉，也可手動切換（手動切換會重設倒數） |
| `ParticleSystem` | 固定大小的物件池，管理雨滴／雪花／龍捲風碎屑／火花四種粒子 |
| `Lightning` | 雷暴天氣的閃電：安全區演算法 + 短暫的閃光與鋸齒狀雷擊路徑 |
| `Tornado` | 龍捲風天氣：鎖定最近的動物、緩慢追逐、漏斗狀繪製 |

背景（天空漸層、雲、視差山丘／相思樹、地面、水坑/積雪疊層、前景草叢）都是獨立的純函式，依 `seasonManager` / `weatherManager` 目前狀態即時算色，不額外持有可變狀態。

## 動物習性實作與季節變化邏輯

每隻動物都是一個 `Animal` 子類別實例，狀態機只有兩三個狀態，靠 `cooldown` 倒數決定「什麼時候該骰一次要不要切換狀態」，而不是每幀都骰，避免行為看起來太雜訊化：

- **獅子**（3 隻）：`WALK ↔ REST`。`REST` 時停在原地、有簡單的尾巴擺動與呼吸感的浮動，`cooldown` 到期後依機率進入休息，乾季機率（0.5）明顯高於濕季（0.25），對應「乾季更趨向群聚」。其中一隻是另一隻的跟隨者（`leader` + `followOffset`），永遠嘗試貼在領頭獅子後方一段距離，但跟隨者自己也有獨立的休息狀態機，所以隊形會鬆散地散開又重新靠攏，而不是死板同步移動。
- **大象**（2 隻）：`WALK ↔ DRINK`。只有靠近水坑（`waterHole.x` 附近 70px 內）時才有機會觸發「喝水」，喝水時軀幹與鼻子會有抬起的動畫。同樣用 leader/follower 模式讓兩隻大象維持成群移動。
- **豹**（2 隻）：`WALK ↔ CLIMB`。靠近場景中兩棵固定的「爬樹點」（`climbTrees`）時才可能觸發攀爬，攀爬時停止水平移動，用 `sin(progress·π)` 算出一個「爬上去再下來」的單一拱形垂直位移，時間到了自動回到 `WALK`。豹沒有跟隨關係，永遠獨立、且基準速度（92–98）明顯高於獅子與大象，移動時額外疊加一個較高頻率的擺盪，讀起來更敏捷。

季節與天氣對行為的影響集中在 `Animal.currentSpeed()`（`baseSpeed × 季節乘數 × 天氣乘數 × 逃跑加成`）與各自 `updateState()` 裡的機率參數，兩層疊加就能讓「乾季 + 暴雨」與「濕季 + 晴天」呈現明顯不同的活動量，不需要為每種組合各寫一套邏輯。

季節：每 60 秒（`SEASON_PERIOD`）在乾／濕之間自動輪替，也可以隨時用按鈕或鍵盤 `S` 手動切換；手動切換會把倒數計時重設為 0，之後的自動輪替從那一刻重新算 60 秒。季節除了影響動物速度與群聚傾向，也直接決定地面／山丘／相思樹／前景草叢的基礎色（乾季偏金黃、濕季偏翠綠）。

## 天氣系統與「雪只在乾季較少出現」

6 種天氣（晴天／下雨／暴雨／下雪／雷暴／龍捲風）共用同一張「基礎天空色（依季節）+ 天氣疊加色（依權重混合）」的表，而不是為 6×2=12 種組合各自手刻一組顏色——`WEATHER_SKY_OVERLAY` 只定義每種天氣的疊加色與混合權重，實際顏色是即時用 `lerp` 算出來的，這樣任何天氣在任何季節下都會有合理但不同的觀感。

「雪只在乾季較少出現」的實作方式：`WeatherManager.spawnRate()` 在天氣為 `snow` 且季節為 `dry` 時，把雪花的生成速率乘上 0.35——使用者仍然可以在乾季手動點選下雪（互動性優先於嚴格寫實），但畫面上的雪明顯稀疏很多，呼應「較少出現」的語意，同時保留 gallery 展示時的可操作性。

## 粒子管理與效能策略（Object Pooling）

`ParticleSystem` 建構時一次配置 450 個粒子物件（純資料物件，不是任何 DOM/Canvas 子物件），全程式生命週期不會再 `new` 出新的粒子：

- `spawn(type, init)` 從一個環狀游標開始找第一個 `active === false` 的 slot 直接重寫，找不到就直接放棄這次生成（維持固定上限，不會無止盡增長）。
- 天氣切換時呼叫 `clearType()` 把「舊天氣對應的粒子類型」整批標成 inactive，讓它們的 slot 立刻可以被新天氣重用。這裡有個小細節：雷暴／下雨／暴雨三種天氣共用同一種粒子類型 `'rain'`（只是初始化的速度與大小不同），所以雷暴切換成普通下雨時**不會**觸發清空——這是刻意的，否則畫面上的雨會在兩種「都在下雨」的天氣之間切換時突兀地消失又重新下起來。
- 粒子的物理更新（`updateParticle`）與繪製（`drawParticle`）都是無狀態的純函式，直接操作物件的欄位，沒有額外配置。
- 主迴圈用顯式 `requestAnimationFrame` + 手動量測的 `dt`（`performance` 時間戳相減），並把單幀 `dt` 限制在 50ms 以內，避免分頁切到背景再切回來時，累積的巨大 `dt` 讓粒子、動物或天氣狀態瞬間跳動。

## 雷暴安全區演算法說明

`Lightning.strike()` 需要保證「閃電永遠不會擊中任何動物」，作法是取樣代替解析解：

1. 在畫面寬度內隨機取 24 個候選 x 座標。
2. 對每個候選，算出它與**所有動物目前 x 位置**的最短距離（`minDist`）。
3. 一路保留「目前看過的候選中，`minDist` 最大的那一個」；只要有候選達到安全門檻 `SAFE_R = 55px` 就提早結束取樣。
4. 用最終選出的 x 座標往下產生一條鋸齒狀路徑（每段加一點隨機水平偏移），畫出雷擊與全螢幕短暫閃光。

因為是「在所有候選裡挑距離最大的」而不是「隨便選一個大於安全半徑的」，即使動物在畫面上分佈得很密集，這個演算法也保證選到當下**相對最安全**的位置——已經用自動化測試連續模擬 20 次落雷，確認每一次與最近動物的距離都 ≥ 55px（詳見下方「驗證方式」）。閃電本身只短暫顯示 0.15 秒（`boltTimer`），不會像有些實作那樣讓雷擊線條一直殘留在畫面上。

## 龍捲風追逐邏輯

`Tornado` 每 0.6 秒重新鎖定一次「目前離自己最近的動物」當作目標（不是每幀都重算，行為更穩定也省運算），然後以固定的 `TORNADO_SPEED = 30`（明顯低於所有動物的基準速度：獅子 50、大象 34、豹 92+）朝目標水平方向緩慢逼近。

動物端則有一個小小的「威脅感知」：當任何動物與龍捲風的距離小於 130px，它的 `fleeBoost` 會平滑（非瞬間）提升到 2.2 倍——這個倍率是刻意調校過的，因為龍捲風天氣本身會把所有動物的移動速度乘上 0.6（見下方天氣速度乘數），如果逃跑加成不夠大，天氣減速最重的大象反而會被恆速的龍捲風追上；調到 2.2 倍後，即使是最慢的大象也能在龍捲風接近時甩開它，讓「追逐」讀起來像貓捉老鼠的動態拉鋸，而不是龍捲風黏在最慢動物身上不放。

## Object Pooling 實作細節

除了上面「粒子管理」一節，物件池的精神也貫穿到動物身上：7 隻動物在整個生命週期內是固定的 7 個物件，離開畫面右側只是把 `x` 重設到左側畫面外的隨機位置（`if (this.x > W + 90) this.x = -Math.random()*300 - 60`），從未被銷毀或重新 `new`。同樣地，粒子池裡的 450 個物件在頁面存續期間只會反覆 `active = true/false` 切換與欄位覆寫，Canvas 2D 的繪製呼叫也都直接讀這些長期存活物件的欄位，長時間執行不會因為短命物件堆積而觸發明顯的 GC 停頓。

## Responsive Embed（320×200 / 16:10）

- HUD（FPS／粒子數／季節倒數／天氣）固定在左上角，8 個天氣＋季節按鈕整合成單一列固定在畫面最下方，兩者用 `@media (max-height:220px), (max-width:360px)` 在極小尺寸下縮小字級與按鈕尺寸。
- 按鈕永遠是 icon-only（用 `title` 屬性提供完整名稱作為 tooltip），8 顆 28px 按鈕即使在 320px 寬也不會溢出或互相重疊。
- 動物車道集中在畫面高度 42%–90% 之間，刻意保留最下方一小段邊界給底部控制列疊加，避免按鈕直接蓋住主要場景內容。
- `resize` 事件 + `ResizeObserver` 雙重保險，畫面尺寸變動時動物的水平位置會等比例換算，垂直車道位置本來就是即時用分數算出，不需要換算。

## 驗證方式（本次提交時做過的測試）

- 用 Playwright 載入頁面，確認 console 沒有錯誤（僅有一個與功能無關的 `favicon.ico` 404）。
- 用程式化測試連續呼叫 `Lightning.strike()` 20 次，確認每次落雷位置與最近動物的距離都 ≥ 55px 安全半徑。
- 用程式化測試讓 `Tornado.update()` 連續執行（模擬 6 秒），確認會鎖定畫面上實際存在的動物並持續朝它靠近。
- 手動切換全部 6 種天氣與 2 種季節，確認粒子數、地面色調、動物速度都有對應變化，且切換當下動物不會停止或瞬移。
- 在 320×200 viewport 下截圖確認主要場景與全部控制項都清楚可見、不重疊。
